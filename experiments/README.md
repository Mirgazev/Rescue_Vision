# Эксперименты по дообучению YOLO11-pose

В этом разделе документированы все экспериментальные прогоны, выполненные в рамках ВКР. Численные значения соответствуют **Таблицам 11–23** Главы 4 ВКР. Все измерения выполнены на одной аппаратной конфигурации, что обеспечивает сопоставимость значений FPS между прогонами.

## 3. Итерации дообучения (Таблица 20 ВКР)

| Версия | Директория прогона | Данные | Эпохи | mAP@0.5 ExDark / RESIDE / COCO | Статус |
|---|---|---|:---:|---|---|
| Default | `yolo11m-pose` | COCO-pretrain Ultralytics | — | 0,7242 / 0,4403 / 0,5204 | базовый уровень |
| **v1** | `runs_finetune/exdark_reside_v12` | ExDark 70 % + RESIDE 20 % + COCO 10 % | до 200 (early stop ≈ 29–59) | 0,8112 / 0,5827 / 0,5225 | первый устойчивый прирост |
| **v2** | `runs_finetune_v2/exdark_fixed` | ExDark 80 % + RESIDE / COCO 20 % | 137 (вручную) | 0,8725 / 0,6654 / 0,5534 | стабильная |
| **v3** | `runs_finetune_v3/exdark_enhance_aware` | + RescueEnhancer на лету (p = 0,50) | 60 | 0,7678 / 0,6265 / 0,5214 | **отклонена** (distribution mismatch) |
| **v4** | `runs_finetune_v4/exdark_enhance_aware_safe2` | безопасная enhance-aware | 25 | **0,8905 / 0,6749 / 0,5646** | **финальная** |

Полные конфигурации каждой версии — в `experiments/configs/v1.yaml`, `v2.yaml`, `v3.yaml`, `v4.yaml`.

### 3.1 Почему v3 отклонена

В v3 алгоритм улучшения изображения применялся к каждому второму обучающему кадру (`enhance_probability = 0,5`) в комбинации с включёнными мозаичной аугментацией (`mosaic = 0,5`) и mixup (`mixup = 0,15`). На стадии валидации (без enhance) произошло падение mAP@0.5 на ExDark с 0,8725 (v2) до 0,7678 — потеря 0,105 в абсолютных единицах. Причина: входное распределение обучения отклонилось от распределения валидации; модель «выучила» сглаженные текстуры, появляющиеся в результате CLAHE и Single-Scale Retinex, и стала хуже работать на кадрах без предобработки. Подробное обсуждение — в подразделе 4.3 ВКР.

### 3.2 Что сделано в v4

Чтобы получить положительный эффект enhance-aware дообучения без переобучения на сглаженных текстурах, в v4 применены три безопасных решения:

1. **Стартовая точка — веса v2**, а не базовая `yolo11m.pt`. Это позволило сохранить уже выученные на ExDark + RESIDE признаки.
2. **Отключены все смешивающие аугментации** (`mosaic = 0`, `mixup = 0`, `copy_paste = 0`). Именно их комбинация с RescueEnhancer была источником distribution mismatch в v3.
3. **Снижена вероятность применения RescueEnhancer** до 0,25 и заморожены первые 22 слоя (вместо 10). Дообучается только head и верхние слои neck.

**Ручная GT-разметка:**

| Режим | mAP@0.5 | Precision | Recall | F1 | FPS |

Метрики авторазметки специально приведены для иллюстрации эффекта data leakage: ground truth получен той же базовой моделью YOLO11m, а тестируется дообученная v4 — отсюда систематически завышенный mAP. Авторазметка используется только для проверки FPS и поведения AUTO-классификатора на потоковом видео, но **не для оценки качества детекции**. Эта оговорка явно сделана в подразделе 4.6 ВКР.

## 5. Качество автоматического классификатора (Таблица 19 ВКР)

Матрица путаницы для пяти классов на 911 размеченных кадрах. Файл: `experiments/results/auto_classifier_confusion_matrix.csv`.

| GT × predict | NIGHT | FOG | SMOKE | RAIN | OFF | Accuracy |
|---|---:|---:|---:|---:|---:|:---:|
| NIGHT | 276 | 0 | 0 | 10 | 14 | 92,0 % |
| FOG | 0 | 160 | 0 | 1 | 0 | 99,4 % |
| SMOKE | 0 | 0 | 147 | 0 | 3 | 98,0 % |
| RAIN | 14 | 0 | 0 | 116 | 20 | 77,3 % |
| OFF | 15 | 0 | 10 | 0 | 125 | 83,3 % |
| **Среднее** | | | | | | **90,5 %** |

Целевой показатель ТЗ 80 % выполнен с запасом 10,5 п.п. Самый слабый класс — RAIN (77,3 %): часть осадковых кадров классифицируется как NIGHT из-за низкого среднего значения яркости. Это поведение является приемлемым: NIGHT-режим в таких кадрах либо безвреден, либо даёт небольшое улучшение, и не приводит к пропуску людей.

## 6. Подбор гиперпараметров inference (Таблицы 21–23 ВКР)

Файл: `experiments/results/hyperparam_grid_conf_imgsz.csv`. Сетка `conf_threshold` × `imgsz` для каждого датасета. Цель — обоснование рабочей точки (`imgsz = 768`, `conf = 0,25`).

**ExDark, mAP@0.5:**

| conf \ imgsz | 640 | 960 | 1280 | 1920 |
|---|---:|---:|---:|---:|
| 0,15 | 0,790 | 0,784 | 0,741 | 0,564 |
| 0,25 | 0,766 | 0,760 | 0,720 | 0,537 |
| 0,35 | 0,739 | 0,735 | 0,697 | 0,518 |
| 0,50 | 0,666 | 0,673 | 0,629 | 0,459 |

Видно, что увеличение разрешения с 768 до 1920 не даёт прироста (на ExDark — даже устойчивое падение из-за роста доли пустых регионов). Этот результат обосновывает использование `imgsz = 768` как рабочей точки для развёртывания.

## 7. Воспроизведение результатов

### 7.1 Получение весов модели

Веса финальной модели v4 (`best.pt`, размер ≈ 39 МБ) не включаются в репозиторий из-за ограничения GitHub. Скачать их можно из раздела [GitHub Releases](https://github.com/Mirgazev/Rescue_Vision/releases) после публикации Release v1.0:

```bash
mkdir -p weights
wget https://github.com/Mirgazev/Rescue_Vision/releases/download/v1.0/best.pt -O weights/best.pt
```

### 7.2 Минимальная команда воспроизведения

Главная цифра ВКР (строка `v4 / OFF` Таблицы 11):

```bash
python3 tools/evaluate_map_fps.py \
    --dataset exdark \
    --root /path/to/ExDark \
    --model weights/best.pt \
    --imgsz 768 --conf 0.25 --iou 0.5 \
    --enhance off --half \
    --output experiments/results/exdark_v4_off.json
```

Результат должен совпасть с приведённой в `exdark_results.csv` строкой с точностью до округления до четвёртого знака. Полные команды для всех таблиц приведены в `docs/usage.md`.

## 8. Главные выводы

1. Целевой mAP@0.5 = 0,77 на основной низкоосвещённой выборке ExDark перевыполнен на 0,1205 в абсолютных единицах (фактически достигнуто 0,8905). Целевой порог 20 FPS превышен во всех потоковых сценариях. Точность автоматического выбора режима 90,5 % превышает требование 80 % на 10,5 п.п.
2. Основной вклад в качество детекции даёт **дообученная модель**, а не runtime-улучшение изображения: на in-distribution данных принудительное NIGHT/FOG/SMOKE снижает mAP на 3–7 п.п. от уровня OFF. На out-of-distribution данных (синтетический дым в SmokeTest) тенденция сохраняется, но количественно слабее.
3. AUTO-классификатор демонстрирует консервативное поведение: на in-distribution и out-of-distribution данных его метрики близки к OFF. В сценариях, где базовая модель не дообучалась под целевое распределение (строки Default из всех таблиц), runtime-улучшение остаётся механизмом адаптации и повышает количественные показатели mAP.

Полная интерпретация этих результатов и их методологические следствия — в подразделах 4.9 и 4.10 ВКР, а также в Заключении работы.
