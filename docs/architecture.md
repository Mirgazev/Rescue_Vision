# Архитектура подсистемы технического зрения

Документ описывает программную архитектуру подсистемы `rescue_vision` и опирается на главу 3 ВКР Миргазева М. А. (РТУ МИРЭА, 2026). Здесь приведены конструктивные решения, обоснование выбора монолитной архитектуры, QoS-профили ROS 2 и схема привязки к целевой платформе Unitree H1.

## 1. Общая структура программной системы

Подсистема построена как **монолитный ROS 2-узел** с последовательным конвейером обработки одного кадра. Логика разделена на пять функциональных блоков:

```
RealSense / rosbag / MP4
        │
        ▼
┌──────────────────────────────────┐
│  ImprovedSceneAnalyzer           │   статистические признаки кадра,
│  (5 признаков + дерево решений + │   иерархическое дерево решений,
│   сглаживание историей 5 кадров) │   правило «3 из 5»
└─────────┬────────────────────────┘
          │ mode ∈ {OFF, NIGHT, FOG, SMOKE, RAIN}
          ▼
┌──────────────────────────────────┐
│   RescueEnhancer                 │   NIGHT: CLAHE + γ + linear stretch
│   (5 режимов с единым            │   FOG:   Dark Channel Prior (½ res)
│    BGR-интерфейсом)              │   SMOKE: Single-Scale Retinex (¼ res)
│                                  │   RAIN:  медианный + билатеральный
└─────────┬────────────────────────┘   OFF:   без обработки
          │ enhanced BGR frame
          ▼
┌──────────────────────────────────┐
│ YOLO11-pose v4 (FP16)            │   дообученные веса, imgsz 768,
│ + ByteTrack                      │   conf 0,25, iou 0,5, class person
└─────────┬────────────────────────┘
          │
          ▼
ROS 2 топики:
  /scene/mode          (SceneMode,        RELIABLE+TRANSIENT_LOCAL)
  /detection/persons   (PersonDetection,  RELIABLE)
  /detection/poses     (Detection2DArray, RELIABLE)
  /vision/debug/image  (sensor_msgs/Image, BEST_EFFORT)
  /vision/performance  (DiagnosticArray,  RELIABLE)
```

## 2. Обоснование монолитной архитектуры

В подразделе 3.6 ВКР рассмотрена альтернатива — разделение конвейера на отдельные ROS 2-узлы (по узлу на каждый функциональный блок). Этот вариант был **отвергнут** по трём причинам:

1. **Накладные расходы на сериализацию.** Передача uint8 BGR-изображения 1280 × 720 между узлами через ROS 2-граф составляет около 2.7 МБ на кадр. При частоте 30 Hz это поток в 80 МБ/с туда-обратно для каждого узла, что критично для бортового компьютера робота Unitree H1.
2. **Конфликт состояния трекера с многопоточностью.** Ultralytics `model.track(persist=True)` хранит состояние внутри объекта и **не является потокобезопасным**. Распределение детекции по нескольким узлам требует либо вынесения трекинга в отдельный узел с очередью, либо отказа от persist=True. Оба варианта снижают качество трекинга.
3. **Сложность отладки.** Монолитный узел даёт единый лог всех этапов конвейера, единый файл конфигурации и единый граф зависимостей. При распределённой архитектуре необходимо синхронизировать переменные между узлами через топики, что усложняет воспроизведение результатов.

Монолитный узел использует `SingleThreadedExecutor` + `MutuallyExclusiveCallbackGroup`, что гарантирует строгую последовательность вызовов callback-методов и совместимость с не-thread-safe библиотеками.

## 3. Источники видеоданных

Узел `rescue_vision_node` подписывается на топик `/camera/color/image_raw` (`sensor_msgs/Image`) и не зависит от конкретного драйвера камеры. На вход поддерживаются три источника:

| Источник | Запуск | Назначение |
|---|---|---|
| Intel RealSense D435i | `pipeline_realsense.launch.py` | штатный запуск на роботе |
| Ранее записанный rosbag | `pipeline_rosbag.launch.py` | повторяемая отладка, экспериментальный прогон |
| Симуляция / image_publisher | `pipeline_sim.launch.py` | проектный шаблон для дальнейшего развития |

## 4. Описание ROS 2-графа

```
                 [realsense2_camera_node]
                          │
                          │ /camera/color/image_raw
                          │   (sensor_msgs/Image, BEST_EFFORT)
                          ▼
              ┌─────────────────────────┐
              │  rescue_vision_node     │
              │  (SceneAnalyzer +       │
              │   RescueEnhancer +      │
              │   YOLO11-pose +         │
              │   ByteTrack)            │
              └────┬───────┬───────┬────┘
                   │       │       │
       /scene/mode │       │       │ /vision/debug/image
       /detection/persons  │       │ /vision/performance
                   │       │       │
                   ▼       ▼       ▼
            [rosbag2     [operator_visualization_node]
             record]      (optional, HUD operator window)
```

* **`camera_node`** (драйвер `realsense2_camera_node`) публикует RGB-кадры с QoS `BEST_EFFORT` и глубиной 1.
* **`rescue_vision_node`** — основной узел подсистемы. Подписан на топик камеры, выполняет полный конвейер обработки и публикует все исходящие топики.
* **`operator_visualization_node`** — опциональный узел оператора. Подписан на debug-изображение, текущий режим сцены и метрики производительности. В штатном режиме на роботе не запускается.
* **`rosbag2 record`** журналирует входной видеопоток и все исходящие топики для последующего анализа.

## 5. QoS-профили

В соответствии с подразделом 3.6 и Таблицей 13 ВКР, для каждого топика выбран QoS-профиль, отвечающий назначению потока данных. Использование разных профилей позволяет одновременно: не терять важные сообщения (детекции, режим), не накапливать очередь устаревших кадров (debug-изображение), и доставлять последнее значение новым подписчикам (режим сцены).

| Топик | Reliability | Durability | History | Depth | Назначение |
|---|---|---|---|---|---|
| `/camera/color/image_raw` | BEST_EFFORT | VOLATILE | KEEP_LAST | 1 | штатный поток камеры — пропуск кадров допустим |
| `/scene/mode` | RELIABLE | TRANSIENT_LOCAL | KEEP_LAST | 1 | редкое изменение — новый подписчик получает последний режим |
| `/detection/persons` | RELIABLE | VOLATILE | KEEP_LAST | 5 | гарантированная доставка детекций |
| `/detection/poses` | RELIABLE | VOLATILE | KEEP_LAST | 5 | гарантированная доставка pose-данных |
| `/vision/debug/image` | BEST_EFFORT | VOLATILE | KEEP_LAST | 1 | визуализация — допустим пропуск |
| `/vision/performance` | RELIABLE | VOLATILE | KEEP_LAST | 5 | диагностика — небольшой буфер для усреднения |

`TRANSIENT_LOCAL` для `/scene/mode` означает, что новый подписчик (например, оператор подключился позже) сразу получит **последнее опубликованное значение режима**, а не будет ждать следующего обновления. Это критично для оператора, чтобы сразу видеть состояние подсистемы.

## 7. Привязка к платформе Unitree H1

Подсистема `rescue_vision` рассматривается как **программный модуль восприятия**, который должен интегрироваться в общий контур мобильной платформы Unitree H1. В рамках настоящей ВКР физическая интеграция и запуск на роботе не выполнялись; пакет подготовлен как проектная схема ROS 2-интеграции.

Низкоуровневое управление движением (стабилизация походки, планирование траектории, управление суставами) отделено от подсистемы восприятия. Узел `rescue_vision_node` **не публикует** команды управления и **не подписан** на состояние суставов. Это разделение повышает безопасность последующей интеграции: ошибка в подсистеме восприятия не может вызвать аварийное движение робота.

В перспективе развития работы предусмотрены два сценария интеграции:

1. **Информационный режим.** Подсистема восприятия публикует детекции и режим в общий ROS 2-граф, оператор видит HUD-окно с разметкой и принимает решения о направлении движения вручную.
2. **Полуавтономный режим.** Дополнительный узел навигации подписывается на `/detection/persons` и `/scene/mode`, формирует команды для контроллера движения Unitree H1 (через стандартный ROS 2-интерфейс производителя). В этом сценарии подсистема `rescue_vision` остаётся независимой и не модифицируется.

## 8. Ограничения текущей версии

В соответствии с разделом «Что НЕ входит в работу» главного README, следующие задачи **вынесены за пределы** настоящей ВКР:

* физический запуск пакета на роботе Unitree H1 не выполнялся;
* физическая или программная симуляция платформы (Gazebo и иные симуляторы) не использовалась;
* низкоуровневые контуры управления Unitree H1 рассматриваются как направления дальнейшего развития работы;
* аппаратная часть мобильной платформы не проектировалась;
* оценка трекинговых метрик (IDF1, MOTA, IDS, FRAG) выполнена только на MOT17 — на видеоматериале МЧС России выполнялась только метрическая оценка детекции и не выполнялась оценка трекинга, так как нет соответствующей разметки.
* MATLAB и его инструментарий в работе не использовались.
